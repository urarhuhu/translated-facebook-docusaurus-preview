# 建立文件

建立一個 Markdown 檔案 `greeting.md`，並將其放置在 `docs` 目錄下。

```bash
website # root directory of your site
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: Create a doc page with rich content.
---

# Hello from Docusaurus

Are you ready to create the documentation site for your open source project?

## Headers

will show up on the table of contents on the upper right

So that your users will know what this page is all about without scrolling down or even without reading too much.

## Only h2 and h3 will be in the TOC by default.

You can configure the TOC heading levels either per-document or in the theme configuration.

The headers are well-spaced so that the hierarchy is clear.

- lists will help you
- present the key points
- that you want your users to remember
  - and you may nest them
    - multiple times
```

:::note

`docs` 目錄下所有以下劃線 (`_`) 為前綴的檔案預設會被視為「部分」頁面並忽略。

詳見[導入部分頁面](../markdown-features/markdown-features-react.mdx#importing-markdown)。

:::

## 文件前言資料 {#doc-front-matter}

[前言資料](../markdown-features/markdown-features-intro.mdx#front-matter)用於為文件頁面提供額外的元數據。前言資料是可選的——Docusaurus 能夠在沒有前言資料的情況下推斷出所有必要的元數據。例如，下文介紹的[文件標籤](#doc-tags)功能需要使用前言資料。所有可能的欄位請參閱[API 文件](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文件標籤 {#doc-tags}

你可以選擇為文件頁面添加標籤，這在[文件側邊欄](./sidebar/index.mdx)之外引入了另一種分類維度。標籤以前言資料中的標籤列表形式傳遞：

```md "your-doc-page.md"
---
id: doc-with-tags
title: A doc with tags
tags:
  - Demo
  - Getting started
---
```

:::tip

標籤也可以透過 `tags: [Demo, Getting started]` 宣告。

詳見所有可能的[Yaml 陣列語法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 組織檔案結構 {#organizing-folder-structure}

Markdown 檔案在 `docs` 資料夾下的排列方式可能對 Docusaurus 內容生成產生多重影響。然而，大多數影響可以與檔案結構解耦。

### 文件 ID {#document-id}

每個文件都有一個唯一的 `id`。預設情況下，文件 `id` 是相對於根 docs 目錄的文件名稱（不包含副檔名）。

例如，`greeting.md` 的 ID 是 `greeting`，而 `guide/hello.md` 的 ID 是 `guide/hello`。

```bash
website # Root directory of your site
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

然而，`id` 的**最後部分**可以由使用者在前言資料中定義。例如，如果 `guide/hello.md` 的內容如下定義，其最終 `id` 為 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

ID 用於在手寫側邊欄或使用文件相關佈局元件或鉤子時引用文件。

### 文件 URL {#doc-urls}

預設情況下，文件的 URL 位置是其相對於 `docs` 資料夾的檔案路徑，但有幾個例外。具體來說，如果檔案命名為以下之一，則檔案名稱不會包含在 URL 中：

- 命名為 `index`（不區分大小寫）：`docs/Guides/index.md`
- 命名為 `README`（不區分大小寫）：`docs/Guides/README.mdx`
- 與父資料夾同名：`docs/Guides/Guides.md`

在所有情況下，預設的 slug 只會是 `/Guides`，不包含 `/index`、`/README` 或重複的 `/Guides` 部分。

:::note

此慣例與[類別索引慣例](./sidebar/autogenerated.mdx#category-index-convention)完全相同。然而，`isCategoryIndex` 配置_不會_影響文件 URL。

:::

使用 `slug` 前言資料來變更文件的 URL。

例如，假設你的網站結構如下：

```bash
website # Root directory of your site
└── docs
    └── guide
        └── hello.md
```

預設情況下，`hello.md` 將可透過 `/docs/guide/hello` 路徑存取。您可以將其 URL 位置變更為 `/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug` 會附加至文件插件的 `routeBasePath`（預設為 `/docs`）。請參閱[僅文件模式](docs-introduction.mdx#docs-only-mode)瞭解如何從 URL 中移除 `/docs` 部分。

:::note

可以使用以下形式：

- 絕對 slug：`slug: /mySlug`、`slug: /`...
- 相對 slug：`slug: mySlug`、`slug: ./../mySlug`...

:::

若希望文件可在根目錄存取，並擁有類似 `https://docusaurus.io/docs/` 的路徑，可使用 slug 前置元資料：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 側邊欄 {#sidebars}

使用[自動生成側邊欄](./sidebar/autogenerated.mdx)時，檔案結構將決定側邊欄結構。

我們對檔案系統組織的建議是：讓您的檔案系統反映側邊欄結構（這樣您就不需要手動編寫 `sidebars.js` 檔案），並使用 `slug` 前置元資料來自訂每個文件的 URL。
