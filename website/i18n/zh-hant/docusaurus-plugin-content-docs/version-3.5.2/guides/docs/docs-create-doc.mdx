# 建立文件

建立一個 Markdown 檔案 `greeting.md`，並將其置於 `docs` 目錄下。

```bash
website # root directory of your site
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: Create a doc page with rich content.
---

# Hello from Docusaurus

Are you ready to create the documentation site for your open source project?

## Headers

will show up on the table of contents on the upper right

So that your users will know what this page is all about without scrolling down or even without reading too much.

## Only h2 and h3 will be in the TOC by default.

You can configure the TOC heading levels either per-document or in the theme configuration.

The headers are well-spaced so that the hierarchy is clear.

- lists will help you
- present the key points
- that you want your users to remember
  - and you may nest them
    - multiple times
```

:::note

`docs` 目錄下所有以下劃線 (`_`) 為前綴的檔案預設會被視為「部分」頁面而忽略。

詳見[導入部分頁面](../markdown-features/markdown-features-react.mdx#importing-markdown)。

:::

## 文件前置元數據 {#doc-front-matter}

[前置元數據](../markdown-features/markdown-features-intro.mdx#front-matter)用於為文件頁面提供額外元數據。前置元數據為可選項——Docusaurus 能在無前置元數據情況下推斷所有必要元數據。例如下文介紹的[文件標籤](#doc-tags)功能需使用前置元數據。完整字段說明請參閱[API 文檔](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文件標籤 {#doc-tags}

標籤需在前置元數據中聲明，為[文件側邊欄](./sidebar/index.mdx)之外提供另一種分類維度。

可定義行內標籤，或引用預先於[`標籤文件`](../../api/plugins/plugin-content-docs.mdx#tags-file)（通常為 `docs/tags.yml`）中聲明的標籤（此為可選項）。

以下範例中：

- `docusaurus` 引用自 `docs/tags.yml` 中預定義的標籤鍵
- `Releases` 為行內標籤，因其不存在於 `docs/tags.yml`

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# Title

Content
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: 'Docs related to the Docusaurus framework'
```

:::tip

標籤亦可透過 `tags: [Demo, Getting started]` 形式聲明。

詳見所有可能的[Yaml 陣列語法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 組織檔案結構 {#organizing-folder-structure}

`docs` 目錄下 Markdown 檔案的排列方式會對 Docusaurus 內容生成產生多重影響，但多數影響可與檔案結構解耦。

### 文件 ID {#document-id}

每份文件皆有唯一 `id`。預設情況下，文件 `id` 為相對於根 docs 目錄的文件名稱（不含副檔名）。

例如 `greeting.md` 的 ID 為 `greeting`，而 `guide/hello.md` 的 ID 為 `guide/hello`。

```bash
website # Root directory of your site
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

但 `id` 的**最後部分**可由使用者在前置元數據中定義。例如若 `guide/hello.md` 的內容如下定義，其最終 `id` 將為 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

ID 用於手動編寫側邊欄時引用文件，或使用文件相關佈局組件與鉤子時。

### 文件 URL {#doc-urls}

預設情況下，文件的 URL 路徑為其相對於 `docs` 目錄的檔案路徑，但有幾項例外。若檔案命名符合以下情況，檔案名稱將不會出現在 URL 中：

- 命名為 `index`（大小寫不敏感）：`docs/Guides/index.md`
- 命名為 `README`（大小寫不敏感）：`docs/Guides/README.mdx`
- 與父目錄同名：`docs/Guides/Guides.md`

上述所有情況中，預設 slug 僅會是 `/Guides`，不會包含 `/index`、`/README` 或重複的 `/Guides` 段落。

:::note

此慣例與[分類索引慣例](./sidebar/autogenerated.mdx#category-index-convention)完全相同。但需注意，`isCategoryIndex`配置項_不會_影響文件URL的生成。

:::

使用`slug`前置元數據可修改文件URL路徑。

例如，假設您的站點結構如下：

```bash
website # Root directory of your site
└── docs
    └── guide
        └── hello.md
```

默認情況下`hello.md`的訪問路徑為`/docs/guide/hello`。您可將其URL修改為`/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug`值會自動附加到文檔插件的`routeBasePath`後（默認為`/docs`）。如需移除URL中的`/docs`前綴，請參閱[純文檔模式](docs-introduction.mdx#docs-only-mode)。

:::note

支持以下兩種slug格式：

- 絕對路徑：`slug: /mySlug`、`slug: /`...
- 相對路徑：`slug: mySlug`、`slug: ./../mySlug`...

:::

若需將文檔設置為根路徑（如`https://docusaurus.io/docs/`），可使用slug前置元數據：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 側邊欄 {#sidebars}

使用[自動生成側邊欄](./sidebar/autogenerated.mdx)時，文件結構將決定側邊欄結構。

我們建議的文件系統組織方式：使文件結構與側邊欄結構保持一致（從而無需手動編寫`sidebars.js`文件），並通過`slug`前置元數據來自定義每個文檔的URL。
