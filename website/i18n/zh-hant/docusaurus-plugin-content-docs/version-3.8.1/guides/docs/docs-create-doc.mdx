# 建立文件

建立一個 Markdown 檔案 `greeting.md`，並將其置於 `docs` 目錄下。

```bash
website # root directory of your site
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: Create a doc page with rich content.
---

# Hello from Docusaurus

Are you ready to create the documentation site for your open source project?

## Headers

will show up on the table of contents on the upper right

So that your users will know what this page is all about without scrolling down or even without reading too much.

## Only h2 and h3 will be in the TOC by default.

You can configure the TOC heading levels either per-document or in the theme configuration.

The headers are well-spaced so that the hierarchy is clear.

- lists will help you
- present the key points
- that you want your users to remember
  - and you may nest them
    - multiple times
```

:::note

`docs` 目錄下所有以下劃線 (`_`) 為前綴的檔案預設會被視為「部分」頁面而忽略。

詳見[導入部分頁面](../markdown-features/markdown-features-react.mdx#importing-markdown)。

:::

## 文件前置元數據 {#doc-front-matter}

[前置元數據](../markdown-features/markdown-features-intro.mdx#front-matter)用於為文件頁面提供額外元數據。前置元數據為可選項——即使沒有前置元數據，Docusaurus 也能推斷出所有必要的元數據。例如，下文介紹的[文件標籤](#doc-tags)功能就需要使用前置元數據。所有可用欄位請參閱[API 文檔](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文件標籤 {#doc-tags}

標籤在前置元數據中聲明，為[文件側邊欄](./sidebar/index.mdx)之外提供了另一種分類維度。

可以定義內聯標籤，或引用預先定義在[`標籤文件`](../../api/plugins/plugin-content-docs.mdx#tags-file)中的標籤（可選，通常為 `docs/tags.yml`）。

以下範例中：

- `docusaurus` 引用的是 `docs/tags.yml` 中預定義的標籤鍵
- `Releases` 是內聯標籤，因為它不存在於 `docs/tags.yml`

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# Title

Content
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: 'Docs related to the Docusaurus framework'
```

:::tip

標籤也可透過 `tags: [Demo, Getting started]` 形式聲明。

詳見所有可能的[Yaml 陣列語法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 組織檔案結構 {#organizing-folder-structure}

Markdown 檔案在 `docs` 目錄下的排列方式會對 Docusaurus 內容生成產生多重影響，但大多數影響可與檔案結構解耦。

### 文件 ID {#document-id}

每份文件都有唯一的 `id`。預設情況下，文件 `id` 是相對於 docs 根目錄的文件名稱（不含副檔名）。

例如，`greeting.md` 的 ID 是 `greeting`，而 `guide/hello.md` 的 ID 是 `guide/hello`。

```bash
website # Root directory of your site
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

但使用者可透過前置元數據定義 `id` 的**最後部分**。例如，若 `guide/hello.md` 的內容如下定義，其最終 `id` 將是 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

ID 用於在手寫側邊欄或使用文件相關佈局元件/鉤子時引用文件。

### 文件 URL {#doc-urls}

預設情況下，文件的 URL 路徑是其相對於 `docs` 目錄的檔案路徑，但有幾項例外。若檔案命名符合以下任一情況，檔案名稱將不會出現在 URL 中：

- 命名為 `index`（不區分大小寫）：`docs/Guides/index.md`
- 命名為 `README`（不區分大小寫）：`docs/Guides/README.mdx`
- 與父目錄同名：`docs/Guides/Guides.md`

以上所有情況中，預設 slug 僅會是 `/Guides`，不會包含 `/index`、`/README` 或重複的 `/Guides` 段落。

:::note

此慣例與[類別索引慣例](./sidebar/autogenerated.mdx#category-index-convention)完全相同。然而，`isCategoryIndex`配置並_不會_影響文件網址。

:::

使用 `slug` 前置元數據來變更文件的網址。

舉例來說，假設您的網站結構如下：

```bash
website # Root directory of your site
└── docs
    └── guide
        └── hello.md
```

預設情況下，`hello.md` 將可透過 `/docs/guide/hello` 存取。您可以將其網址變更為 `/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug` 將會附加至文件插件的 `routeBasePath`（預設為 `/docs`）。請參閱[僅文件模式](docs-introduction.mdx#docs-only-mode)以了解如何從網址中移除 `/docs` 部分。

:::note

可以使用以下形式：

- 絕對 slug：`slug: /mySlug`、`slug: /`...
- 相對 slug：`slug: mySlug`、`slug: ./../mySlug`...

:::

若您希望文件可在根目錄存取，並擁有類似 `https://docusaurus.io/docs/` 的網址路徑，可使用 slug 前置元數據：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 側邊欄 {#sidebars}

當使用[自動生成側邊欄](./sidebar/autogenerated.mdx)時，檔案結構將決定側邊欄結構。

我們對於檔案系統組織的建議是：讓您的檔案系統反映側邊欄結構（這樣您就不需要手動編寫 `sidebars.js` 檔案），並使用 `slug` 前置元數據來自訂每個文件的網址。
