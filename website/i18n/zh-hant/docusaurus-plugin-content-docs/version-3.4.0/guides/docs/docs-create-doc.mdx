# 建立文件

建立一個 Markdown 檔案 `greeting.md`，並將其置於 `docs` 目錄下。

```bash
website # root directory of your site
├── docs
│   └── greeting.md
├── src
│   └── pages
├── docusaurus.config.js
├── ...
```

```md
---
description: Create a doc page with rich content.
---

# Hello from Docusaurus

Are you ready to create the documentation site for your open source project?

## Headers

will show up on the table of contents on the upper right

So that your users will know what this page is all about without scrolling down or even without reading too much.

## Only h2 and h3 will be in the TOC by default.

You can configure the TOC heading levels either per-document or in the theme configuration.

The headers are well-spaced so that the hierarchy is clear.

- lists will help you
- present the key points
- that you want your users to remember
  - and you may nest them
    - multiple times
```

:::note

`docs` 目錄下所有以下劃線 (`_`) 為前綴的檔案都會被視為「部分」頁面，預設情況下會被忽略。

詳閱[導入部分頁面](../markdown-features/markdown-features-react.mdx#importing-markdown)了解更多。

:::

## 文件前置資料 {#doc-front-matter}

[前置資料](../markdown-features/markdown-features-intro.mdx#front-matter)用於為文件頁面提供額外的元數據。前置資料是可選的——Docusaurus 能夠在沒有前置資料的情況下推斷出所有必要的元數據。例如，下文介紹的[文件標籤](#doc-tags)功能就需要使用前置資料。關於所有可能的欄位，請參閱[API 文件](../../api/plugins/plugin-content-docs.mdx#markdown-front-matter)。

## 文件標籤 {#doc-tags}

標籤在前置資料中宣告，為[文件側邊欄](./sidebar/index.mdx)之外引入了另一種分類維度。

可以定義內聯標籤，或引用在[`標籤文件`](../../api/plugins/plugin-content-docs.mdx#tags-file)（通常為 `docs/tags.yml`，可選）中預先定義的標籤。

在以下範例中：

- `docusaurus` 引用在 `docs/tags.yml` 中宣告的預定義標籤鍵
- `Releases` 是一個內聯標籤，因為它不存在於 `docs/tags.yml` 中

```md title="docs/my-doc.md"
---
tags:
  - Releases
  - docusaurus
---

# Title

Content
```

```yml title="docs/tags.yml"
docusaurus:
  label: 'Docusaurus'
  permalink: '/docusaurus'
  description: 'Docs related to the Docusaurus framework'
```

:::tip

標籤也可以透過 `tags: [Demo, Getting started]` 宣告。

詳閱所有可能的[Yaml 陣列語法](https://www.w3schools.io/file/yaml-arrays/)。

:::

## 組織檔案結構 {#organizing-folder-structure}

Markdown 檔案在 `docs` 資料夾下的排列方式可能對 Docusaurus 內容生成產生多重影響。然而，大多數影響可以與檔案結構解耦。

### 文件 ID {#document-id}

每個文件都有一個唯一的 `id`。預設情況下，文件 `id` 是相對於根 docs 目錄的文件名稱（不包含副檔名）。

例如，`greeting.md` 的 ID 是 `greeting`，而 `guide/hello.md` 的 ID 是 `guide/hello`。

```bash
website # Root directory of your site
└── docs
   ├── greeting.md
   └── guide
      └── hello.md
```

然而，`id` 的**最後部分**可以由使用者在前置資料中定義。例如，如果 `guide/hello.md` 的內容如下定義，其最終 `id` 為 `guide/part1`。

```md
---
id: part1
---

Lorem ipsum
```

ID 用於在手動編寫側邊欄時引用文件，或在使用與文件相關的佈局元件或鉤子時使用。

### 文件 URL {#doc-urls}

預設情況下，文件的 URL 位置是其相對於 `docs` 資料夾的檔案路徑，但有幾個例外。具體來說，如果檔案命名為以下之一，檔案名稱將不會包含在 URL 中：

- 命名為 `index`（不區分大小寫）：`docs/Guides/index.md`
- 命名為 `README`（不區分大小寫）：`docs/Guides/README.mdx`
- 與父資料夾同名：`docs/Guides/Guides.md`

在所有情況下，預設的 slug 僅為 `/Guides`，不包含 `/index`、`/README` 或重複的 `/Guides` 部分。

:::note

此慣例與[類別索引慣例](./sidebar/autogenerated.mdx#category-index-convention)完全相同。但需注意，`isCategoryIndex`配置項_不會_影響文件URL。

:::

使用 `slug` 前置元數據來變更文件的URL路徑。

例如，假設您的站點結構如下：

```bash
website # Root directory of your site
└── docs
    └── guide
        └── hello.md
```

預設情況下 `hello.md` 的訪問路徑將是 `/docs/guide/hello`。您可以將其URL修改為 `/docs/bonjour`：

```md
---
slug: /bonjour
---

Lorem ipsum
```

`slug` 值會附加到文檔插件的 `routeBasePath` 基礎路徑（默認為 `/docs`）。關於如何從URL中移除 `/docs` 部分，請參閱[純文檔模式](docs-introduction.mdx#docs-only-mode)。

:::note

可使用以下形式：

- 絕對路徑：`slug: /mySlug`、`slug: /`...
- 相對路徑：`slug: mySlug`、`slug: ./../mySlug`...

:::

若需將文檔設置在根路徑（如 `https://docusaurus.io/docs/`），可使用 slug 前置元數據：

```md
---
id: my-home-doc
slug: /
---

Lorem ipsum
```

### 側邊欄 {#sidebars}

使用[自動生成側邊欄](./sidebar/autogenerated.mdx)時，文件結構將決定側邊欄結構。

我們對文件系統組織的建議是：使文件結構與側邊欄結構保持一致（這樣就無需手動編寫 `sidebars.js` 文件），並通過 `slug` 前置元數據來自定義每個文檔的URL。
